Part 1: Ray Generation and Scene Intersection

First, we set up the min_t and max_t for the ray, which we take from the camera. Then we find the x and y bounds of our screen, which are determined from the hFov and vFov. Then we calculate the direction of the ray using the change in x, y, and z, which is 1 by default. Then, we set the ray's position to be the camera's position and give it the sampled direction, transformed with the c2w matrix. This generates a ray. Second, to illuminate a pixel, we sample randomly within a pixel for a chosen amount of samples, then average our results to get the radiance for that pixel. When we add triangles and spheres to a scene, we need to be able to calculate the first intersection of a ray with an object. For intersections with triangles, we implement the Möller–Trumbore algorithm, which efficiently calculates the time at which a ray intersects a triangle. The algorithm also provides the barycentric coordinates of the intersection, allowing for smooth interpolation of the vertex normals. For spheres, we follow an algorithm that uses the quadratic formula to calculate the up to 2 intersections, and take the one that occurs first since the second is blocked. With all of this put together, we can render simple objects in the Cornell box.

Part 2: Bounding Volume Hierarchy

For our BVH construction algorithm, we implemented a recursive method that consisted of counting the number of primitives from start to end and first checking if they were over a max leaf size. If so, then we recursed down by splitting the primitives into two partitions then calling the contruction algorithm on those partitions. If not, then we hit a base case and the node created just stored those primitives. To split the primitives, we found the longest axis of the bounding box surrounding all of the primitives, then splitting at the midpoint of that side of the bounding box. We used the centroid of the primitives' bounding boxes to split them into the two partitions.

For CBlucy.dae, render time without BVH acceleration is 147.2730 seconds. With BVH acceleration, rendering takes 0.0536 seconds. For maxplank.dae, render time without BVH acceleration is 35.7926 seconds, whereas with acceleration it only takes 0.0570 seconds. While CBlucy.dae has 133796 primitives and maxplank.dae has 50801 primitives, their render times with BVH acceleration are very similar. This shows the log(N) speedup from N with the naive algorithm well.

Part 3: Direct Illumination

First, we implement direct lighting by sampling uniformly on a hemisphere. We select random directions on the hemisphere above the intersection point, and for each direction, we check to see if the ray will intersect any objects in the BVH. If the ray does not intersect any objects, then we calculate the contribution of that ray to the pixel's radiance by multiplying the emission, BSDF, and cosine of the angle between the ray and the surface normal (given by the z coordinate of our sample). We average the contributions from all sampled rays to get an estimate of the direct illumination at that point. In the second option, we only sample directions that point towards the light source. We calculate the direction from the intersection point to the light source, and check if that ray intersects any objects in the BVH. If it does not, then we calculate the contribution of that ray to the pixel's radiance using the same formula as before. There is an added check that the intersection t must be less than the distance to the light source, since it's okay to be behind the light. We can also use the z coordinate of the sample in the object's coordinate system to see if the light is behind the object, in which case we can skip the ray intersection test and contribution calculation since the light won't contribute to the pixel's radiance.
After implementing direct illumination, we can see that the hemisphere lighting results in a noisier image than the light source sampling, since many of the rays sampled in the hemisphere do not contribute to the pixel's radiance. Specifically, we noticed a small difference in noise for the pectoral muscles of the beast, which are in soft shadow. Overall, hemisphere lighting is more computationally expensive than light source sampling, since it requires more rays to get a good estimate of the direct illumination. Lighting sampling is more efficient since it focuses sampling efforts in directions that are likely to contribute to the pixel's radiance, resulting in a less noisy image with fewer samples. One benefit is that both methods are unbiased, which gives them good convergence properties. However, light source sampling is more efficient in scenes with small or distant light sources, while hemisphere lighting can be more effective in scenes with large or nearby light sources that are likely to be hit.

Part 4: Global Illumination

In order to implement global illumination, we need to allow light to bounce off of more than one surface before hitting the camera. The base case of this recursion is when we reach max depth, at which point we return. Otherwise, we sample a new ray direction based on the BSDF of the surface at the intersection point, get the contribution of that ray to the radiance. This ray has one fewer depth, so we are making progress to the max depth. Depending on whether the total is accumulated, we either return the last contribution or the contribution plus the recursive call to the new ray.
In both cases, soft shadows show up more clearly with global illumination, since light can bounce off of other surfaces and illuminate areas that are in shadow with direct illumination.

Part 5: Adaptive Sampling

To implement adaptive sampling, as we are accumulating samples for a pixel, we keep track of the mean and variance of the irradiance of the pixel so far. If the expression \( I = 1.96 \times \sigma/\sqrt{n} \) is less than the threshold (\(\mu \times m\) where \(m\) is the max tolerance (default is 0.05)), we stop sampling for that pixel. We check the inequality above every sampleCountBuffer samples, where sampleCountBuffer is a parameter that determines how often we check the stopping condition.
The purpose of adaptive sampling is to reduce the number of samples taken for pixels that have already converged to a good estimate of their radiance, while still taking enough samples for pixels that are noisy and have not yet converged. We can use a higher number of samples per pixel for the adaptive sampling method, since it will stop early for pixels that have already converged, resulting in a more efficient rendering process.
